How it works

Give one manager your goals. It runs the team.

The General topic is the orchestrator's office. Give it plain-language goals — no commands — and it splits the work, hires a worker per task, opens a visible thread for each, briefs them, supervises at checkpoints, and reports back.

1

Give a goal

Tell the orchestrator what you want in plain language — "fix the login 500, then audit deps". It plans the work and hires the workers.

2

It hires workers

One worker per task, each on its own branch in an isolated git worktree and its own visible thread. Same-repo parallel work never collides.

3

Watch & micromanage

Every worker thread streams live. Step into any of them — both the worker and the orchestrator hear you. Nothing happens behind your back.

Workers run headless with bypassPermissions on your chosen backend — Claude Code by default, or Codex. The orchestrator is woken only at checkpoints (done, blocked, needs a decision, you interjected), never per token.

Features

An org you can actually see into.

Accurate to the box you run — nothing more, nothing hidden.

Any surface

Drive the same org from Telegram, a browser, or a VS Code panel — one shared client. Slack is next; nothing in the core knows which.

Glass-walled delegation

Every worker conversation is a visible thread you can watch and interject into — both the worker and the orchestrator see your message.

Isolated git worktrees

Each worker gets its own branch in its own worktree — same-repo parallel work is safe, and un-landed work is never deleted, only reported.

Checkpoint supervision

Workers run autonomously; the orchestrator is woken only at meaningful checkpoints — done, blocked, needs a decision — never per token. Wakes are coalesced.

Any backend

Workers run on Claude Code by default, or Codex with one env var. The orchestrator, supervision, and isolation don't change — a worker is a worker.

Self-hosted & resumable

Runs anywhere Docker does with restart: unless-stopped; sessions persist and reconnect across restarts. You own the box and the data — no lock-in.

Drive it

Plain language in General. A few commands when you want them.

Just talk to the orchestrator — or reach for a command. (The web & VS Code surfaces use a thread list and a message box instead of slash-commands.)

# General the orchestrator's office
(plain message)A goal for the orchestrator — it hires and briefs the workers.
/new <path>A direct, orchestrator-less session in that repo.
/listEvery thread — orchestrator, workers, direct — and its status.
/setupCheck the group is configured right (admin, topics, allowlist).
# worker thread watch & steer
(any message)Steers that worker — and the orchestrator sees it too.
(photo / file)Handed to the session; images ride as vision input.
/stopInterrupt the current turn.
/killEnd the session; a clean worktree is removed, a dirty one kept.

First run? DM the bot /whoami to get your id — no third-party bot needed — add it to the allowlist, and you're the boss.
